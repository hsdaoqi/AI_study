# Part 1. 先分清:预训练 / SFT / 强化学习,是三种不同的"教法"

大模型从"啥也不会"到"能对话"要过三关。用**学生备考**类比:

| 阶段 | 干啥 | 学生类比 | 你学过没 |
|---|---|---|---|
| **预训练 (Pretrain)** | 海量文本上学"预测下一个词" | 把全世界的书读一遍,学会语言本身 | 了解即可 |
| **SFT (监督微调)** | 用"问题→标准答案"对教它怎么回答 | **抄范文**:照着标准答案学格式、学套路 | ✅ 你做过(`train_sft.py`) |
| **RL (强化学习)** | 让它自己答、给答案打分、按分数改进 | **自己做题被老师打分**,再针对性改进 | ← **现在这关** |

**为什么 SFT 之后还要 RL?**(主讲/理解都要答得上)

- SFT 是**模仿**:只能学会"范文里有的东西",范文没覆盖的就不会,而且它不知道"哪个答案更好",只知道"照抄"。
- RL 是**优化**:给一个**目标(reward)**,让模型**自己探索**怎么答能拿高分,**可以超过范文的水平**。
- 典型收益:让回答更符合人类偏好(RLHF)、让数学/代码**答对率**上升(可验证 reward,DeepSeek-R1 那套)、让格式更规范。

> 一句话:**SFT 教它"像样地说话",RL 教它"把话说对/说好"。** 你这次要跑的 GRPO,就是 RL 这一关里现在最主流的方法。
>
> "**文本**强化学习" = 就是在**纯文本 LLM** 上做 RL(相对于多模态/图像 RL),别被这个词唬住,就是本文讲的这套。

<a id="part-2"></a>
# Part 2. 强化学习的基本盘(用打游戏理解)

RL 不是 LLM 专属,它是一套通用框架。先用**打游戏 / 训练小狗**理解这几个词,Part 3 再套到 LLM。

## 一句话定义

> **强化学习 = 一个"智能体"在"环境"里不断"做动作",根据拿到的"奖励"调整自己,目标是让长期总奖励最大。**

## 六个核心词(打游戏版)

| 术语 | 打游戏类比 | 一句话 |
|---|---|---|
| **智能体 / Agent** | 你操控的角色 | 做决策的主体 |
| **策略 / Policy** | 你的"操作习惯/打法" | 看到局面,决定做什么动作的**规则**(在 LLM 里 = **模型本身**) |
| **环境 / Environment** | 游戏世界 | 智能体所处、会对动作做出反应的东西 |
| **状态 / State** | 当前画面 | 此刻的局面 |
| **动作 / Action** | 按了哪个键 | 智能体做的选择 |
| **奖励 / Reward** | 得分 / 掉血 | 环境给的反馈信号(好=正,坏=负) |

## 再加三个"进阶但关键"的词

| 术语 | 类比 | 一句话 |
|---|---|---|
| **轨迹 / Trajectory(也叫 Episode)** | 从开局到结束的**一整局** | 一连串"状态→动作→奖励"的完整过程 |
| **回报 / Return** | 这一局的**总得分** | 一条轨迹上奖励的总和 |
| **优势 / Advantage** | 这一步比"平均水平"**好多少** | 某个动作相对基准的好坏(GRPO 的核心量,Part 6 细讲) |

## 目标

> 调整**策略**(打法),让**期望回报**(平均总得分)最高。"调整策略" = 更新模型参数。

**RollOut(下一节主角)** 就藏在这里:要知道"当前打法好不好",你得**先照当前打法实际打几局看看得分**——那"打几局"的过程,就是 rollout。

<a id="part-3"></a>
# Part 3. 把这些术语套到 LLM 上,到底对应什么 ⭐

RL 套到"训练一个语言模型"上,每个词具体变成:

| RL 通用术语 | 在 LLM 里具体是什么 |
|---|---|
| **策略 Policy** | **就是这个大模型**(它决定"给定上文,下一个词输出啥") |
| **状态 State** | 当前的 prompt + 已经生成的部分 |
| **动作 Action** | 生成**下一个 token**(一整条回答 = 一串动作) |
| **轨迹 / Episode** | **一个完整的回答**(从 prompt 到生成结束) |
| **奖励 Reward** | 给这个**完整回答**打的分(对不对、好不好、格式对不对) |
| **RollOut** | 模型对着 prompt,**采样生成若干个完整回答**(把策略"跑"出来) |
| **优势 Advantage** | 这个回答比"同题其它回答的平均"好多少 |

**关键差异(和打游戏不同的一点)**:LLM 的 RL 通常是"**一句话回答 = 一整局**",奖励往往在**回答结束时**才给一个总分(不是每个 token 都有分)。这叫**稀疏/终局奖励**,GRPO 就是专门处理这种情况的好方法。

<a id="part-4"></a>
# Part 4. RollOut 详解 ⭐(那个你没见过的新词)

## 什么是 RollOut

> **RollOut = 让当前模型(策略)对着一批 prompt,实际生成出一批完整回答。** "把策略跑一遍、演出来"。

名字来源:RL 里管"让策略在环境里跑出一条完整轨迹"叫 roll out a trajectory(把轨迹"铺展开")。在 LLM 里,它朴素地就等于**推理生成**——只不过这些生成结果不是给用户看的,而是**拿去打分、拿去训练**的。

**学生备考类比**:老师要评估学生现在的水平,得让学生**先把题实际做一遍**(还每题做几种解法)。这个"让学生把题做出来"的动作,就是 rollout。做出来的答卷,就是待打分、待学习的材料。

## 在 GRPO 里,RollOut 长这样

**每个 prompt 生成 G 个回答**(G 叫 `num_generations`,常见 8 或 16):

```
prompt: "求 12 × 13"
  ├─ rollout 回答1: "12×13 = 12×10 + 12×3 = 120+36 = 156"   ✓
  ├─ rollout 回答2: "12×13 = 155"                            ✗
  ├─ rollout 回答3: "= 156,因为..."                          ✓
  ├─ ...
  └─ rollout 回答8: "大概是 150 左右"                          ✗
```
- 为什么要生成**多个**?因为 GRPO 要**同题横向比**(组内相对),没有多个就没法比(Part 6)。
- 生成时用**采样**(temperature>0),这样 8 个回答各不相同,才有比较的意义——这就用到你 vLLM 文档 Part 4 学的采样参数。

## 为什么 RollOut 是 RL 训练里最贵的一步 ⭐(闭环到 vLLM)

算一下:每步训练取 64 个 prompt、每个 rollout 8 个回答、每个回答 512 token → **一步就要生成 64×8×512 ≈ 26 万个 token**。而且**每训练一步都要重新 rollout 一次**(因为模型变了,得用新模型重新生成)。

> **所以 RL 训练里,一大半时间花在"生成"上,不是"反向传播"上。** 谁的生成快,谁的 RL 就快。

**这就是 vLLM 登场的地方**:RL 框架都用高吞吐推理引擎来做 rollout。**SWIFT 默认用 vLLM 做 rollout**——你上周装的 vLLM,正是这里的发动机。整条任务链(vLLM→ReAct→RL)在此咬合:vLLM 不只是"部署推理快",它是 **RL 训练的加速核心**。

## RollOut 的两种部署方式(多卡时要选,Part 8 再展开)

| 模式 | 意思 | 适合 |
|---|---|---|
| **colocate(同居)** | rollout 的 vLLM 和训练**挤在同一批 GPU 上**,轮流用 | 卡少(2 卡起步 demo) |
| **server(分离)** | 单独起一个 vLLM 服务占几张卡专门 rollout,训练占另几张 | 卡多、要吞吐 |

<a id="part-5"></a>
# Part 5. Reward 详解(分数从哪来)

RollOut 生成了一堆回答,**谁来打分?** 这就是 **Reward(奖励)**。它是 RL 的"指挥棒"——你把 reward 设成什么,模型最后就往什么方向进化。**Reward 设计是 RL 里最需要动脑的地方**。

## 两大类 reward

| 类型 | 怎么打分 | 例子 | 谁在用 |
|---|---|---|---|
| **奖励模型 / Reward Model (RM)** | **另训一个模型**来给回答打分(学人类偏好) | "这个回答更礼貌/更有用" | RLHF(ChatGPT 那套) |
| **可验证 / 规则 reward (Verifiable)** | **写死的函数/规则**判分,客观可查 | 数学答案对不对、代码能否运行、格式对不对 | GRPO / DeepSeek-R1 / 你要跑的 demo |

你的 demo 用**可验证 reward**(简单、不用另训 RM、最适合入门)。常见几个 reward 函数:

- **accuracy reward(答对率)**:答案和标准答案一致 → +1,否则 0。(数学/选择题)
- **format reward(格式分)**:是否按要求把思考放 `<think>...</think>`、答案放 `\boxed{}` → 符合 +1。
- **length reward**:控制别太长太短。
- **多个 reward 加权相加** = 总 reward(比如 `总分 = 1.0×答对 + 0.2×格式`)。

> R1 的关键洞察:**只用"答对没有"这种可验证 reward,配 GRPO,就能逼出模型的推理能力**(它为了答对,自己学会了"多想几步")。你的 demo 就是这个思路的最小版。

# Part 6. GRPO 详解 ⭐(在你已有笔记上讲深)

你 `GRPO笔记` 已经懂了核心("答 8 次、比组内平均、无 critic、KL 缰绳")。这里把它讲进**训练细节**,并和 PPO 说清区别。

## 先回顾:GRPO 一句话

> **GRPO(Group Relative Policy Optimization,组相对策略优化)= 同一道题生成一组(G 个)答案,用"组内平均分"当基准,比平均好的答案就提高它的概率、比平均差的就降低,再用 KL 拴住别跑偏。**

## 核心:优势(Advantage)怎么算 —— 组内标准化

RollOut 给一道题生成了 G 个答案,reward 分别是 r₁…r_G。每个答案的**优势**:

```
       r_i − (这一组的平均分)
A_i = ────────────────────────
          (这一组的标准差)
```

- **减去平均**:高于组内平均 → A 为正(该鼓励);低于 → A 为负(该压制)。
- **除以标准差**:归一化,让不同题的优势可比、训练更稳。

**学生类比**:同一道题,你写了 8 种解法,老师给这 8 种打分。**不看绝对分,只看"你这版比这 8 版的平均高还是低"**——高的记住多用,低的以后少用。这就是"组相对"。

## GRPO vs PPO:省在哪(主讲高频问)

PPO(另一个经典 RL 算法)要算优势,得**再训一个"价值网络(critic/value model)"**去估计"这道题大概能得几分"当基准。这多一个和主模型一样大的网络,**费显存、费算力、还难训**。

> **GRPO 的聪明:不训 critic,直接用"组内 G 个答案的平均分"当基准。** 省掉一整个大网络 → 显存/实现都轻一半 → **特别适合大模型**。这就是 DeepSeek 用它的原因。

| | PPO | GRPO |
|---|---|---|
| 基准(baseline)怎么来 | 单独训一个 value 网络估 | **组内平均**,不用额外网络 |
| 额外开销 | 多一个大模型 | 只多"每题生成 G 个" |
| 适合 | 通用 RL | **大模型 + 可验证 reward** |

## 目标函数(想看公式的看这里,不看不影响用)

GRPO 最大化(简化版,逐行翻译):

```
         1   G   1                   π_θ(答案i)                          
J = E [ ─── Σ  ─────  Σ   min(  ───────────── · A_i ,  clip(比值,1±ε)·A_i ) ]  − β·KL(π_θ ‖ π_ref)
         G  i=1 |o_i|  t         π_old(答案i)                          
```
- `π_θ / π_old`:新模型对这个答案的概率 / 旧模型的概率(**比值**)。比值×优势 = "往优势正的方向多推"。
- `min(..., clip(..., 1±ε))`:**别一步推太猛**(信任域裁剪,ε 常 0.2),防训崩。
- `A_i`:上面算的组内优势。正的答案→提高其概率;负的→降低。
- `− β·KL(π_θ ‖ π_ref)`:**KL 缰绳**。π_ref = 训练前的原始模型(**reference model,参考模型**,全程冻结)。这项惩罚"新模型偏离原模型太远",防止它为了刷分学得魔怔(比如只会输出 `\boxed{156}` 不说人话)。β 控制缰绳松紧。

> **reference model(参考模型)** = 训练开始时那个模型的**冻结副本**,专门用来算 KL、当"别忘本"的锚。这就是为什么 RL 训练显存里要同时装**两个模型**(可训练的 π_θ + 冻结的 π_ref),比 SFT 更吃卡(Part 8)。

# Part 7. 合起来:一个完整的 RL 训练循环长啥样 ⭐

把 RollOut(Part4)+ Reward(Part5)+ GRPO(Part6)串成一圈。这是整份文档的总装图:

```
┌─ 一个训练 step ────────────────────────────────────────────────┐
│                                                                 │
│  1. 取一批 prompt(比如 64 道题)                                  │
│         ↓                                                       │
│  2. RollOut:当前模型对每道题采样生成 G=8 个回答  ← vLLM 加速这步     │
│         ↓         (64×8 = 512 个回答)                            │
│  3. Reward:每个回答打分(答对率+格式分)→ 得到 512 个 reward         │
│         ↓                                                       │
│  4. 算优势:每道题内部,8 个回答各自 (r−组均值)/组标准差 → A_i        │
│         ↓                                                       │
│  5. GRPO loss:提高高优势回答的概率、降低低优势的,− β·KL(拴住参考模型)│
│         ↓                                                       │
│  6. 反向传播,更新模型参数一点点                                    │
│                                                                 │
└──────────────────── 回到 1,用更新后的模型再来一轮 ─────────────────┘

盯什么信号 = 训练好不好:reward 平均分【随 step 缓慢上涨】= 模型在变强 ✓
```

对照你会的 SFT:SFT 是"给答案→算 loss→更新"(1 步搞定);**RL 多了"自己生成答案(RollOut)+ 打分(Reward)+ 组内比较(GRPO)"这前几步**,数据是模型**自己产的**,不是喂的。这就是"强化学习"和"监督学习"的根本区别。

# Part 8. 多卡:为什么 RL 比 SFT 更吃卡,怎么配

## 为什么 RL 特别吃显存(所以任务要"多卡")

一步 GRPO,显存里要同时住着:
1. **可训练的策略模型 π_θ**(+ 优化器状态,SFT 也有)
2. **冻结的参考模型 π_ref**(算 KL 用)← SFT 没有,RL 多的
3. **rollout 的 vLLM 引擎**(生成用,还要 KV cache)← SFT 没有,RL 多的
4. 一批 G 倍数量的回答的激活值 ← 比 SFT 多 G 倍数据

**所以同样的模型,RL 比 SFT 吃卡得多**,单卡经常装不下 → 任务才要求**多卡**。

## 多卡怎么分工(接上你会的 DDP / Accelerate)

- **训练部分**:用**数据并行**(你学过的 DDP/Accelerate)——每张卡放一份模型,不同卡处理不同 prompt,梯度 all-reduce 同步。SWIFT 底层就是用 `torchrun`/accelerate 拉起多进程(和你 `train_sft.py` 一个道理)。
- **RollOut 部分**:两种放法(Part 4 提过):
  - **colocate(同居)**:vLLM 和训练**共享同一批卡**轮流用。卡少(2 卡 demo)就用这个。
  - **server(分离)**:单独拿几张卡起 vLLM 服务专做 rollout,训练用另几张。卡多、追吞吐用这个。

> 你 demo 阶段:**2 张卡 + colocate 模式**最省事。命令里通过 `NPROC_PER_NODE=2` + `CUDA_VISIBLE_DEVICES=0,1` 指定用两张卡(和你 `torchrun --nproc_per_node=2` 同一个意思)。

# Part 9. SWIFT 框架 + 跑通 GRPO demo(实操)

## SWIFT 是什么

**ms-swift**(魔搭 ModelScope 出品)= **一站式训练/微调/RL 框架**。SFT、DPO、PPO、GRPO、多模态全支持,**一条命令行**起训,你不用手写训练循环。地位:它之于"训练/RL",就像 vLLM 之于"推理"——**把脏活封装好,你只调参数**。

## 装(给 RL 单独的环境,别和别的混)

```bash
# 可以复用你的 vllm 环境(SWIFT 的 GRPO 正好要 vLLM),也可另开一个
pip install ms-swift -U
pip install math_verify   # 数学答案校验(accuracy reward 要用)
```

## 跑一个 GRPO demo(数学题 + 可验证 reward)

一条命令的模板(**每行都对应前面讲的概念**,右侧注释):

```bash
CUDA_VISIBLE_DEVICES=0,1 \
NPROC_PER_NODE=2 \
swift rlhf \
    --rlhf_type grpo \                         # 用 GRPO(Part 6)
    --model Qwen/Qwen2.5-3B-Instruct \         # 被训练的策略模型 π_θ
    --dataset AI-MO/NuminaMath-TIR \           # 数学题数据集(有标准答案好算 reward)
    --reward_funcs accuracy format \           # 奖励=答对率+格式分(Part 5)
    --num_generations 8 \                      # 每题 rollout G=8 个回答(Part 4)
    --use_vllm true \                          # 用 vLLM 做 rollout(Part 4 闭环)
    --vllm_mode colocate \                     # 2卡同居模式(Part 8)
    --max_completion_length 1024 \             # 每个回答最多生成多少 token
    --per_device_train_batch_size 2 \          # 每卡每步几个 prompt
    --learning_rate 1e-6 \                     # RL 学习率通常很小
    --beta 0.04 \                              # KL 缰绳松紧 β(Part 6)
    --num_train_epochs 1 \
    --output_dir output/grpo_demo
```

> ⚠️ **flag 名会随 ms-swift 版本微调**(它更新很快)。跑之前先 `swift rlhf --help` 看当前版本的准确参数名,或查 ms-swift 官方 GRPO 文档。**跑起来报错就把报错贴给我,和我们清 vLLM 那些坑一样,逐个改。**

## 怎么算"跑通了"

- 日志里 reward / completion 相关指标**随 step 往上走**(哪怕慢),就说明在学 → **任务"跑通 demo 即可"达成**。

# Part 10. 卡点速查

| 现象 | 原因 | 解决 |
|---|---|---|
| `CUDA out of memory`(RL 特别容易) | 策略+参考+vLLM 三份挤爆显存 | 换更小模型(1.5B/3B)、调小 `num_generations`、`per_device_train_batch_size`、`max_completion_length`、给 vLLM 调低显存比例 |
| 装 ms-swift 后 torch/vllm 冲突 | 版本不匹配 | 单开环境;按 ms-swift 官方要求的版本装 |
| reward 一直不涨 / 训崩(NaN) | 学习率太大、β 太小学飞、reward 设计有问题 | 调小 lr(1e-6 级)、加大 β、先确认 reward 函数打分正确 |
| flag 报 `unrecognized arguments` | 版本对不上 | `swift rlhf --help` 查当前 flag 名 |
| rollout 很慢 | 没开 vLLM,或 colocate 抢显存 | `--use_vllm true`;卡够就换 server 模式 |
| 数据集/模型下载失败 | 直连外网 | 用魔搭源(SWIFT 默认走 ModelScope,一般没问题);必要时 `export HF_ENDPOINT=https://hf-mirror.com` |

<a id="part-12"></a>
# Part 12. 术语表(所有名词一句话)

| 术语 | 一句话 |
|---|---|
| **强化学习 / RL** | 让模型自己做、给结果打分、按分改进的训练法(区别于"抄范文"的 SFT) |
| **文本强化学习** | 在纯文本 LLM 上做 RL(相对多模态) |
| **策略 / Policy** | 决定"看到上文输出什么"的规则;在 LLM 里就是模型本身 |
| **动作 / Action** | 生成的下一个 token(一整条回答=一串动作) |
| **奖励 / Reward** | 给回答打的分,RL 的指挥棒 |
| **轨迹 / Episode** | 一个完整回答(从 prompt 到生成结束) |
| **优势 / Advantage** | 某回答比"同题平均"好多少;GRPO 里 = 组内标准化的 reward |
| **RollOut** | 让当前模型对 prompt 实际采样生成一批回答("把策略跑一遍");RL 里最耗时,用 vLLM 加速 |
| **num_generations (G)** | 每个 prompt rollout 几个回答(GRPO 组大小,常 8) |
| **Reward Model (RM)** | 另训一个模型来打分(RLHF 用) |
| **可验证 reward** | 用规则/函数客观判分(答对率、格式、能否运行);GRPO/R1 常用 |
| **GRPO** | 组相对策略优化:同题生成一组、用组内平均当基准、无需 critic、加 KL 约束 |
| **PPO** | 经典 RL 算法,要额外训一个 value/critic 网络当基准(GRPO 省掉了它) |
| **critic / value 网络** | 估"这题大概能得几分"当基准的网络;GRPO 不用它 |
| **reference model / π_ref** | 训练前模型的冻结副本,算 KL、防跑偏的锚 |
| **KL 散度 / KL 缰绳** | 惩罚新模型偏离参考模型太远,防止学魔怔;β 控松紧 |
| **信任域裁剪 / clip (ε)** | 每步别把概率推太猛,防训崩(常 ε=0.2) |
| **colocate / server** | rollout 的 vLLM 与训练"同居一批卡" / "分开占卡"两种部署 |
| **SWIFT / ms-swift** | 魔搭的一站式训练/微调/RL 框架,命令行起 GRPO |
| **RLHF** | 用人类偏好训 reward model 再做 RL(ChatGPT 那套);GRPO 常改用可验证 reward |

---

> 配套:概念齐了就去 Part 9 跑 demo;跑通标准 = 多卡上 reward 会涨。
> 承上启下:RollOut 用的是 vLLM(`VLLM从零到测速`);GRPO 核心接你 `GRPO笔记`;RL 是 `train_sft.py` 那套 SFT 的下一站。三份连起来,你就把"训练→推理→RL"一条线全打通了。
